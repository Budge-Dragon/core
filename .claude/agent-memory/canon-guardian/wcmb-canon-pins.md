Plan-mode canon rulings for **W-CMB** (attack resolution, per-class stat derivation, skill damage, kill-reward loop), to re-apply when the code-mode review runs. Every fact below was verified term-by-term against `docs/reference/openmu-facts/0…md` (stats) and `6…md` (damage/exp) and `5…md` (drops), and the divergences were reproduced with a Python script (fractions module).

**Why:** Two rulings correct the wave's own DoD numbers and are non-obvious; substantiated, not asserted.

**How to apply (verify each in the implementation):**

- **Integer derivation scheme = POOLED SINGLE-DIVIDE per stat, NOT per-term truncation.** OpenMU aggregates each derived stat as ONE float sum `Σ(coeff·stat)` and truncates once when consumed as an integer pool. The faithful + fixed-point-canonical (Q-format: one scale, round once at boundary) reproduction is: for each derived stat express as `(Σ integer-numerator terms)/D` over a common denominator D, accumulate the numerator in a widened `u64`, divide once by a `NonZero` D, truncate toward zero (= floor; all quantities non-negative). The wave header ALREADY prescribes this for DL max_health `(97+3L+4V)/2` — the rule must be applied CONSISTENTLY to every multi-fractional-term stat. **Per-term truncation (floor each `coeff·stat` then sum) is the non-canonical bespoke variant** — it rounds N times, biases downward up to N−1 sub-units, and DIVERGES from OpenMU. Verified divergences: MG max_ability (`.15·24+.3·40+.25·30+.2·60=35.1`) → per-term 34, faithful 35; DL max_ability (`=35.45`) → per-term 34, faithful 35. **The DoD/scenario expected values "MG ag 34" and "DL ag 34" are WRONG — the faithful canon values are 35/35.** Also diverges generically (not caught by the chosen examples): attack_rate `5L+1.5A+0.25S=(20L+6A+S)/4` for odd-Agi + Str≡2,3 mod 4 (e.g. A41 S2 → per-term 61, pooled 62); Elf max_mana `6+1.5E+1.5L` for odd E and L; MG/DL phys span (two fractional terms); DL attack_rate `5L+2.5A+S/6+C/10` (three fractional). `scale_ratio(value,num,den)` stays valid for SINGLE-term stats and the damage/exp multipliers; multi-fractional-term stats must use the pooled numerator, never summed per-term scale_ratio calls.
  - Nested floor-division is unambiguous: `floor(floor(x/a)/b)==floor(x/(a·b))`, so DefenseFinal=`Agi/6` etc. carry no ambiguity. Only SUMS of ≥2 fractional terms diverge.
  - DL max_mana `38+1.5(E−15)+L`: `E−15` must not underflow — DL base energy is 15 so total energy ≥15 is a domain floor; rely on the proven floor (total stat ≥ class base), do NOT `saturating_sub` defensively.

- **Elemental application curve: ADD a faithful `1/(resistance+1)` primitive; do NOT reuse `roll_resistance`.** `services::chance::roll_resistance` is `uniform_below(255) < byte`, i.e. P(resisted)=byte/255, so apply-iff-!resist = `(255−r)/255` — a LINEAR-complement curve. The sourced apply probability (facts 6 L94) is the RECIPROCAL `1/(r+1)`. They agree only at the boundaries (R=0 always apply, R=255 immune) which are the ONLY cases the DoD scenarios test; mid-range is off by ~24× (R=10: 0.96 vs 0.09). This is NOT an approximation. Canonical faithful form (Bernoulli via uniform integer, reusing the `rng::uniform_below` seam): `apply = (r < 255) && (uniform_below(r+1) == 0)`. `r+1∈1..=256` → NonZeroU32 via the `NonZeroU32::MIN.saturating_add(r)` idiom, no unwrap. `r<255` is the immune gate (a genuine domain branch, not a should-never-happen). Gate BOTH the ailment-set AND the lightning shove on this one application roll (lightning displacement is gated on the lightning-element application SUCCEEDING, per facts, not merely on the hit landing). Keep `roll_resistance` if it models a separate linear resistance save; it is simply the wrong primitive here.
