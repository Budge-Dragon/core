Plan-mode canon rulings for **W-EFFECT** (ActiveEffect store, apply/advance services, effective_profile fold, iced/frozen movement seam), to re-apply in code mode. Verified against `docs/reference/openmu-facts/3…md` (magic effects), `6…md` (constants), `0…md` (PoisonDamageMultiplier). Tick base = 50ms (confirmed in units.rs tests).

**Why:** Three DoD/spec numbers are internally CONTRADICTORY and one canon value is under-specified; the rulings are load-bearing and non-obvious.

**How to apply (verify each in the implementation):**

- **Durations/magnitudes — ALL verified authentic (reference 3 L110-123, 6 L114/131), integer-exact at 50ms base.** Iced 10s→Tick200 ×1/2 speed; Frozen 5s→Tick100 no-move; DefenseReduction 10s→Tick200 defense ×9/10; Defense(DK ShieldSkill 200) 4s→Tick80 incoming ×1/2; GreaterDamage 60s→Tick1200 +3+E/7 (=13@E70); GreaterDefense 60s→Tick1200 +2+E/8 (=10@E70); Heal instant +5+E/5 (=19@E70, saturating). All floor-division integer; ×1/2 and ×9/10 via scale_ratio (num/den), NO floats. `DurationMs::in_ticks` (div_ceil) gives all these exactly (no rounding, all divide evenly).

- **POISON DAMAGE BASIS RULING = 3% of the TARGET's MAXIMUM health per tick, resolved at apply, stored constant.** Reference under-specifies (PoisonDamageMultiplier=0.03 is an ATTACKER attribute, basis not stated). Ruled target-max-HP because: (a) attacker-side small-fraction "multiplier" semantics, (b) classic MU poison = %-of-victim-max-HP, (c) the "no attacker" e2e forces a TARGET-based basis, (d) the DoD "constant per-tick amount" DEFINITIVELY rules out a CURRENT-health basis (which would decay each tick). `per_tick = scale_ratio(target_max_health, 3, nonzero(100))`, baked at apply, never re-read. Flag W-SRC: byte-verify vs OpenMU poison DoT source (independent of extraction); flag the low-HP floor (3% of maxHP<34 floors to 0 → poison deals 0; consider a ≥1 min-floor).

- **POISON e2e "dies to DoT from FULL health in ≤7 ticks" is UNACHIEVABLE under the 3%-of-maxHP ruling and MUST be reworked.** 7 × 3% = 21% of max HP total — never lethal from full; and floors to 0 for max-HP < 34. Fix the e2e: pre-damage the monster below 21% (needs an attacker/seeded sub-full), OR assert non-lethal monotonic decrease + exact 21%-of-max total. The "poison tick reaches zero → StopByDeath clears all" mechanic is still correct; only the from-FULL death expectation is wrong.

- **POISON lifetime is governed by the remaining-tick COUNTER (7), self-terminating on the 7th tick at Tick(420) — NOT by the nominal 20s/Tick(400) gate.** 7 ticks at 60,120,…,420 (first at 60 per "no tick before 60"); 7×3s = 21s > 20s. A 400-tick removal gate truncates the 7th tick → only 6 fire. The DoD line "Poison expiry Tick(400)" is INCONSISTENT with the 7-ticks-at-60..420 scenarios: poison's ActiveEffect must carry next_tick + remaining-count and EffectExpired co-fires with the 7th PoisonTick at 420; do NOT store a 400 expiry that gates poison removal. Catch-up on gap is bounded by the counter (loop while remaining>0 && next_tick.reached(now)).

- **DECAY (3-tick/10s poison) is S3+ Summoner content → OUT of pre-S3 scope.** Decay is a Summoner curse skill; Summoner = Season 3 (reference 3 L229 excludes all Summoner skills/effects); not in the 075/095d skill lists (3 L217). Removing Decay makes pre-S3 Poisoned a SINGLE uniform profile (7 ticks/20s→420) and DISSOLVES the "7-vs-3 info-loss" problem — TargetHit.inflicted's bare Ailment::Poisoned maps to one profile, no count threaded, no fabricated value. Drop the Decay scenarios/bullets.

- **STACKING = SubType-replace (reference 3 L68): same SubType → new one REMOVES the existing, refreshing duration.** Iced & Frozen BOTH map to SubType 254 (255−Ice) → they are MUTUALLY EXCLUSIVE in canon (applying one clears the other). The plan's per-IDENTITY replace model (Iced replaces only Iced) is canon-correct for every other effect (each owns its SubType) but LOSES the Iced↔Frozen cross-replacement — it would let an entity be Iced AND Frozen simultaneously. NON-CANONICAL unless the Ice pair shares one replace-slot OR the deviation is documented (Frozen's no-move dominates Iced's half-move, so coexistence isn't game-breaking, but it isn't faithful). Frozen's source (Ice Arrow) is 095d+/S6-dataset — a backport, note provenance.

- **CombatBonus-as-effect-currency = sound canonical choice (its own doc names "effect" as a producer; verified first consumer).** Additive effects map cleanly (GreaterDefense→Defense{amount}; GreaterDamage→see below). `IncomingDamagePct` already exists ("folds multiplicatively") for Defense-buff ×1/2 — but CombatProfile has NO incoming field and resolve_attack never reads one (grow ONE defender-side incoming-reduction field, apply as final defender step in resolve_attack, no new param). DefenseReduction ×9/10 does NOT fit additive DefensePct (Percent 0..=100 INCREASE) — needs a reduction variant or an effective_profile derivation step; NEVER smuggle a multiplier into DefensePct, never resolve ×9/10 to a stale apply-time integer (base defense can move).

- **GreaterDamage fold — pin the pipeline slot (SUGGESTION, balance-data not canon-structure).** DoD says "raise the physical SPAN by 13" (Min+Max). But folding into span-MAX makes Critical(=max) and Excellent(=6/5·max) AMPLIFY the +13 (excellent → +15.6). If OpenMU adds GreaterDamageBonus as a FLAT FINAL additive (CombatBonus::Damage{amount}) it is NOT amplified. Decide/source which; they diverge observably at crit/excellent tiers. Not a blocker (exact damage-pipeline placement is domain data), but the CombatBonus variant chosen encodes the decision.

**Verified clean (no change):** apply/advance pure, RNG-free, tick-driven (now:Tick in, no clock); absolute Ticks resolved at apply, advance takes no TickDuration; ActiveEffects serde-plain on both entities; unified total skill router (Damaging|ApplicableBuff|Heal|Deferred) folding the existing `classify` (Option) — the DamagingSkill-subset precedent; ApplicableBuff = {Defense,GreaterDamage,GreaterDefense}+Heal, party/InfiniteArrow/Alcohol/Summon/Teleport/NovaCharge/RecallParty route onward via or-patterns; Pool::restored = saturating_add(amount).min(max) sibling of `reduced`; mobility disposition Free/Slowed(×1/2 Fixed)/Immobilized SUPPLIED to movement (resolve_step takes speed:Fixed; Frozen short-circuits AI) — movement stays effect-unaware; effective_profile transient per-strike, never persisted; poison apply reads TARGET max-HP (not caster stats), buff apply reads caster Energy. HP/MP/AG regen (RecoveryInterval 3000ms, reference 0 L81) correctly deferred — distinct concern needing W-INV stat rates.
